# 缠论规则规格

这份文档用于约束本项目第一阶段的缠论识别口径。目标不是覆盖所有流派，而是先确定一套可实现、可测试、可复盘的规则，作为 Python 程序的统一标准。

## 1. 范围

本版本先定义以下内容：

- K 线基础数据格式
- 预处理和包含关系处理
- 分型识别
- 笔识别
- 线段识别
- 中枢识别

本版本暂不细化以下内容：

- 趋势、盘整、背驰的完整自动识别细则
- 严格意义上的一二三类买卖点全部细则
- 多级别联立分析
- 自动交易执行

但本版本会先把严格理论口径写清楚；文档中凡明确标注为“严格口径”的定义，代表项目后续应收敛到的统一标准，不等于当前代码已经全部实现。

## 2. 统一数据约定

### 2.1 K 线字段

每根 K 线至少包含以下字段：

- `ts`: 时间戳，统一为 pandas `datetime64[ns]`
- `open`: 开盘价
- `high`: 最高价
- `low`: 最低价
- `close`: 收盘价
- `volume`: 成交量，可选但建议保留

### 2.2 顺序要求

- 数据必须按时间升序排列
- 不允许重复时间戳
- 不允许 `high < low`
- 缺失值必须在预处理阶段被剔除或修复

### 2.3 分析级别

第一阶段默认一次只分析单一级别数据，例如：

- 日线
- 30 分钟线
- 5 分钟线

不同级别不在同一次主流程中混用。

## 3. 预处理规则

### 3.1 基础清洗

进入缠论识别前，必须完成：

- 时间排序
- 重复值去重
- 无效 K 线过滤
- 必要字段类型标准化

### 3.2 包含关系处理

第一阶段采用“先去包含，再识别分型”的方案。

这里的“处理方向”仅用于包含关系合并，和后续识别出的笔方向不是同一概念。
后续某一笔被确认，不能反向决定前面某段包含关系应按向上还是向下合并。

定义相邻两根 K 线 `A`、`B`：

- 若 `A.high > B.high` 且 `A.low < B.low`，则 A 严格包含 B
- 若 `A.high < B.high` 且 `A.low > B.low`，则 B 严格包含 A

注意：

- 相等高点不算包含
- 相等低点不算包含
- 只有同时满足“高点更高、低点更低”的严格包络，才做去包含合并

当出现包含关系时，使用方向相关的合并规则生成一根标准化 K 线：

- 若当前处理方向为向上，则合并后：
  - `high = max(A.high, B.high)`
  - `low = max(A.low, B.low)`
- 若当前处理方向为向下，则合并后：
  - `high = min(A.high, B.high)`
  - `low = min(A.low, B.low)`

其中“当前处理方向”定义为最近两根非包含标准化 K 线的方向：

- 若后一根的 `high` 和 `low` 都更高，方向为向上
- 若后一根的 `high` 和 `low` 都更低，方向为向下

若序列起始阶段无法判断方向，则先保留，直到出现第一个明确方向后再继续处理。

严格口径补充：

- 方向必须先确定，再执行包含合并；不能在方向未知时默认按向上或向下处理
- 一段连续包含链一旦已确定方向，在该包含链结束前应保持该方向不变
- 仅当后续出现新的“非包含标准化 K 线”组合时，才允许重新判定方向
- 不能因为“当前 bar 包含前一临时 bar”就直接把方向翻转
- 若一段包含链起始时方向未知，应先暂存该链，直到出现首个可判方向的非包含组合，再回头按该方向完成该链的合并

### 3.3 标准化 K 线输出

去包含后得到 `normalized_bars`，后续分型、笔、线段全部基于这个序列，而不是原始 K 线序列。

每根标准化 K 线需要保留：

- 合并后的 `high`、`low`
- 起止时间范围
- 对应原始 K 线索引列表

这样后续做可视化和解释时可以回溯到原始数据。

## 4. 分型定义

### 4.1 顶分型

标准化 K 线序列中，若第 `i` 根相对左右两侧都满足：

- `high[i] >= high[side]`
- `low[i] >= low[side]`
- 且至少一项严格更大

则第 `i` 根构成顶分型。

### 4.2 底分型

标准化 K 线序列中，若第 `i` 根相对左右两侧都满足：

- `high[i] <= high[side]`
- `low[i] <= low[side]`
- 且至少一项严格更小

则第 `i` 根构成底分型。

这意味着：

- “等高但低点更高”仍可构成有效顶分型
- “等低但高点更低”仍可构成有效底分型
- 例如 `03-23 / 03-24 / 03-25` 在不构成包含时，可把 `03-24` 识别为有效底分型

### 4.3 分型最小窗口

- 分型必须基于连续 3 根标准化 K 线判断
- 首尾两根不能构成分型

### 4.4 分型去重规则

若连续出现多个同类分型候选：

- 连续多个顶分型，只保留最高的那个
- 连续多个底分型，只保留最低的那个

“最高”按分型中心 K 线的 `high` 比较，“最低”按分型中心 K 线的 `low` 比较。

## 5. 笔定义

### 5.1 笔的构成

一笔由相邻且类型相反的两个有效分型构成：

- 顶分型到后续底分型，形成向下一笔
- 底分型到后续顶分型，形成向上一笔

### 5.2 成笔条件

第一阶段采用较稳妥的“分型确认 + 最小间隔”规则：

- 起点和终点必须是类型相反的两个有效分型
- 两个分型之间至少间隔 1 根独立标准化 K 线
- 笔内不能被更新的更强同类分型替代

严格端点约束：

- 向上笔中，终点顶分型的价格端点必须有效突破起点底分型三根窗口的最高点
- 向下笔中，终点底分型的价格端点必须有效跌破起点顶分型三根窗口的最低点

其中“三根窗口”指起点分型对应的左、中、右三根标准化 K 线；若处于边界无法完整取三根，则按实际可取窗口计算。

更具体地说：

- 若底分型之后先出现顶分型，则先形成候选向上一笔
- 若在该候选笔确认前又出现更高的顶分型，则用更高顶分型替代旧终点
- 若顶分型之后先出现底分型，则先形成候选向下一笔
- 若在该候选笔确认前又出现更低的底分型，则用更低底分型替代旧终点

### 5.3 笔的确认

当一个候选笔的终点分型已经成立，并且其后出现了反向分型候选时，前一笔被确认。

换句话说，笔允许在末端随行情延伸被更新，但一旦后续反向结构出现，上一笔即冻结。

### 5.4 笔的属性

每一笔至少输出：

- `direction`: `up` 或 `down`
- `start_fx_id`
- `end_fx_id`
- `start_ts`
- `end_ts`
- `high`
- `low`
- 覆盖的标准化 K 线区间

### 5.5 严格确认口径（本项目当前实现）

为了避免“伪反向笔”，本项目使用更严格的确认流程：

- 笔的识别顺序从左到右，起点分型先形成“候选笔”
- 候选向下笔：从顶分型向后找底分型，且中心索引至少间隔 3（两个三K分型窗口不重叠，且中间至少保留 1 根独立标准化 K 线）
- 候选向上笔：从底分型向后找顶分型，且中心索引至少间隔 3
- 候选向下笔的终点底分型，必须跌破起点顶分型三K窗口最低点
- 候选向上笔的终点顶分型，必须突破起点底分型三K窗口最高点
- 若候选笔未被后续有效反向笔确认，则该笔继续延伸（同向更极值分型替代终点）

有效反向确认规则：

- 当前候选笔终点之后，首先出现的反向分型进入“确认候选”
- 若该确认候选间隔不足，则不能直接确认上一笔
- 若后续出现同类且更强的反向分型，可以替代前一个确认候选
- 只有这个“最强确认候选”本身满足最小间隔时，上一笔才确认
- 若先出现了间隔不足但更极值的反向分型，而后面只有更弱的同类反向分型，则后一笔不能反向补确认

分型确认依赖笔确认：

- 只有作为“已确认笔”端点出现的分型，才视为确认分型
- 因此分型与笔存在联动确认关系，不应把全部三K分型都直接当成有效分型

## 6. 线段定义

### 6.1 线段输入

线段基于“已确认的笔”序列计算，不直接从 K 线计算。

### 6.2 基本规则

第一阶段采用简化且工程上稳定的定义：

- 线段由至少 3 笔构成
- 相邻笔方向必须交替
- 一条线段必须由一组具有连续推进特征的笔组成

### 6.3 段内特征

向上线段通常表现为：

- 关键高点不断抬高
- 关键低点整体抬高

向下线段通常表现为：

- 关键低点不断降低
- 关键高点整体降低

### 6.4 线段终结

第一阶段采用“被反向破坏”原则：

- 若当前向上线段的结构高低点推进被后续笔有效破坏，则该线段结束
- 若当前向下线段的结构高低点推进被后续笔有效破坏，则该线段结束

这里的“有效破坏”在第一阶段具体化为：

- 对向上线段，后续向下笔跌破该线段最近一个关键低点
- 对向下线段，后续向上笔突破该线段最近一个关键高点

### 6.5 实现建议

线段是最容易出现流派分歧的部分。第一版实现时，建议先保持如下策略：

- 先实现一个可运行的简化版本
- 所有线段拐点都保留中间推导数据
- 用样例图手工比对后再继续迭代

### 6.6 当前实现口径

当前项目中的 `src/chanlun/segment.py` 采用的是“工程化简化线段”规则，具体约束如下：

如需一份更适合人工核图的短说明，见 `docs/segment-implementation-guide.md`。

- 线段识别函数可接收全量笔序列，但实现内部只使用 `is_confirmed=True` 的已确认笔
- 任意未确认尾笔都不参与线段起段、扩展或终结判定
- 起段需要连续 3 笔，且方向必须交替，首尾笔同向
- 向上线段起段要求第 3 笔高点高于第 1 笔高点；向下线段起段要求第 3 笔低点低于第 1 笔低点
- 段内扩展按“两笔一组”推进：先出现一笔反向回撤，再出现一笔同向创新高/新低，才允许线段继续延伸
- 向上线段若后续反向笔跌破最近关键低点，则前一线段确认终结；向下线段若后续反向笔突破最近关键高点，则前一线段确认终结
- 若尾部没有出现有效破坏，但已经满足起段与推进条件，则保留一个 `is_confirmed=False` 的未确认尾段

这一口径是当前图表导出的实际行为，应视为当前实现标准；若后续要向更严格缠论段定义收敛，应在此基础上单独升级，而不是让文档与代码继续分叉。

## 7. 中枢定义

### 7.1 中枢输入

严格定义中，中枢应基于“某级别走势类型”识别。

第一阶段为了工程实现，暂以“已确认笔”作为次级别走势类型的近似代理。

因此本节需要区分两层口径：

- 严格口径：依据缠论原文思想，先区分进入段、中枢内部三段、离开段
- 工程近似：在暂不实现完整走势类型拆分前，用已确认笔近似表达次级别走势类型

### 7.2 最小中枢

按缠论原文思想，某级别走势中枢由至少 3 个连续次级别走势类型的重叠部分构成。

若下跌走势进入某一震荡区间，则最先向下离开原走势的那一笔，更适合作为“进入笔”看待，而不直接计为中枢内部第 1 笔。

在笔级近似口径下，应按以下严格方式识别最小中枢：

- 先识别一笔进入笔，用于说明走势进入某一潜在中枢区域
- 从进入笔之后的反向回拉笔开始，记为中枢内部第 1 笔
- 此后至少还需要连续 2 笔，方向交替，共形成连续 3 笔
- 这 3 笔的价格区间必须存在公共重叠，才可认定为最小中枢

也就是说，不能简单把“任意连续 3 笔重叠”都直接当作中枢；若首笔只是进入笔，则应从其后的反向笔开始重新计数。

设中枢内部这 3 笔的区间分别为：

- 第 1 笔区间 `[low1, high1]`
- 第 2 笔区间 `[low2, high2]`
- 第 3 笔区间 `[low3, high3]`

则中枢区间定义为：

- `zs_low = max(low1, low2, low3)`
- `zs_high = min(high1, high2, high3)`

若 `zs_low < zs_high`，则认为存在有效重叠中枢。

严格口径下还应满足：

- 这 3 笔必须是进入笔之后的连续 3 笔
- 相邻笔方向必须交替
- 不应把进入笔本身直接计入中枢内部三笔

例如：若一段下跌后先出现一笔反弹、再下探、再反弹，这后面三笔才是中枢内部的候选三笔；前面的下跌笔应视为进入笔。

### 7.3 中枢扩展

若后续笔继续与当前中枢区间发生重叠，则该中枢可以扩展。

扩展时：

- 中枢的参与笔列表增加
- 中枢重叠区间重新计算
- 若新重叠区间消失，则当前中枢终结

扩展时仍应维持“进入笔不计入中枢内部三笔”的原则；扩展针对的是已经成立的中枢本体，而不是回头把进入笔并入中枢。

### 7.4 中枢属性

每个中枢至少输出：

- `start_bi_id`
- `end_bi_id`
- `zs_low`
- `zs_high`
- `peak_high`
- `peak_low`
- 参与笔列表

### 7.5 当前实现状态

当前项目代码中的中枢识别仍采用简化口径：

- 使用已确认笔作为输入
- 从左到右扫描候选区间
- 识别扩展后的中枢并去除被其完全包含的子中枢

但当前实现尚未完全落实本节 7.2 的严格“进入笔不计入中枢内部三笔”口径。

因此：

- 文档中的本节定义，代表后续应收敛到的严格规则
- 若当前图表结果与本节不一致，应以本节规则为准，后续再修正实现

## 8. 走势类型与背驰定义

本节补充的是严格理论定义，用于统一术语、图表表达和后续实现目标。

其中要特别区分两层含义：

- 严格口径：对应缠论原文层面的走势类型、走势分解、走势终完美、趋势背驰与盘整背驰定义
- 工程口径：当前项目是否已经把这些定义完整自动化，是另一件事

### 8.1 基本术语

- 走势：打开某一级别走势图后看到的价格运动整体
- 走势类型：该级别上的上涨、下跌、盘整三类完成形态
- 趋势：走势类型中的上涨与下跌
- 盘整：走势类型中不属于上涨或下跌趋势的完成形态

这里的“走势”与“走势类型”不能混用：

- 走势是连续展开中的客观过程，既包含已经完成的部分，也包含正在进行的部分
- 走势类型是对某一段完成结构的分类结果

### 8.2 盘整与趋势的严格定义

在任一级别上：

- 某完成的走势类型只包含 1 个该级别走势中枢，称为该级别盘整
- 某完成的走势类型至少包含 2 个依次同向的该级别走势中枢，称为该级别趋势
- 该方向向上时称上涨趋势，向下时称下跌趋势

趋势中的同级别中枢之间必须满足：

- 方向一致
- 彼此依次推进
- 中枢区间之间不存在重叠

因此：

- “只有一个中枢”是盘整与趋势的第一条分界线
- “至少两个同向且彼此不重叠的中枢”是趋势成立的必要条件
- 若只有一个中枢后向外运动，但该走势类型尚未完成，不能提前把它确认为趋势

### 8.3 走势终完美

本项目采用如下严格表述：

- 任何级别的任何走势类型终要完成
- 趋势终要完成
- 盘整也终要完成

“走势终完美”在文档中只表达两层含义：

- 任一走势类型不可能永远处于逻辑上未完成状态
- 一种走势类型完成后，必然转化为其他走势类型的展开

这里不能把“终要完成”误写成“立刻反转”或“马上结束”。
一个走势类型在满足最小结构后仍然可能继续延伸，因此“可完成”和“已完成”不是同一件事。

### 8.4 走势结构分解

本项目采用以下两个基础定理作为严格口径：

- 任何级别的任何走势，都可以分解成同级别上涨、下跌、盘整三种走势类型的连接
- 任何级别的任何走势类型，都至少由 3 段以上次级别走势类型构成

落到工程表达时，应把“走势”和“走势类型连接”分开描述：

- 整个观察窗口内的走势，是若干同级别走势类型的连接
- 单个走势类型内部，又必须能继续分解为不少于 3 段次级别走势类型

因此后续无论做图、做 JSON，还是写解释文本，都不能把一段尚未完成的局部波动直接当成独立走势类型输出。

### 8.5 走势类型延伸

为了避免把延伸误判成完成，本项目先统一以下严格口径：

- 盘整在出现 3 段连续次级别走势类型并形成中枢后，已经具备随时完成的条件
- 但只要围绕该中枢继续延伸且未完成类型转换，仍应视为同一个盘整走势类型的延伸
- 趋势在形成 2 个依次同向的同级别中枢后，也已经具备随时完成的条件
- 但只要新的同向中枢继续生成，仍应视为同一个趋势走势类型的延伸

延伸的识别关键不是“已经走了多久”，而是：

- 盘整是否仍围绕原有结构展开，且未确认转换成趋势或其他盘整连接
- 趋势是否仍在生成新的同向中枢，且未确认转换成盘整或反向趋势

### 8.6 完成、延伸与转化示例

为了避免把“走势类型连接”和“一个走势类型内部延伸”混淆，后续统一使用如下记号：

- 小写 `a`、`b`、`c`：同级别走势类型之间的连接段或离开段，级别低于所讨论的主级别走势类型
- 大写 `A`、`B`、`C`：所讨论主级别上的中枢
- 带 `~` 的记号：表示对已发生部分重新组合后，把它视为一个更高一层含义下的已知局部

#### 8.6.1 标准趋势完成例子

若某级别下跌走势可写成：

```text
a + A + b + B + c
```

并满足：

- `A`、`B` 是同级别中枢
- `B` 相对 `A` 依次同向推进
- `A` 与 `B` 之间没有重叠
- `c` 为最后一个离开段，且该走势整体最终完成

则这是一段标准的该级别下跌趋势。

这里的关键不是字母形式本身，而是：

- 两个同级别中枢已经成立
- 中枢之间满足趋势要求
- 整段结构作为一个完整走势类型已经完成

同理，向上趋势也可以写成同样形式，只是方向相反。

#### 8.6.2 “下跌 + 盘整” 的完成连接

若一段 5 分钟下跌走势已经完成，随后又走出一个 30 分钟盘整走势，那么其语义是：

```text
下跌(完成) + 盘整(完成)
```

这表示：

- 前面的下跌已经是一个完成的走势类型
- 后面的盘整也是另一个完成的走势类型
- 两者是两个已完成走势类型的连接

此时不能再把后面的盘整回写成前面下跌内部的一部分，否则会把“类型连接”误写成“同一走势的延伸”。

#### 8.6.3 最后一个中枢扩展时，不是“下跌 + 盘整”

仍以上面的：

```text
a + A + b + B + c
```

为例。若 `B + c` 没有独立完成后再接出一个新的完成盘整，而是在运行中直接发生中枢扩展，例如把原先按 5 分钟理解的最后一个中枢继续扩成 30 分钟中枢，那么其语义不是：

```text
5分钟下跌 + 30分钟盘整
```

而是：

```text
a~ + A~
```

其中：

- `a~` 表示前面已经能确认的一段已知局部
- `A~` 表示由原 `B + c` 继续扩展出来的新中枢结构

这时更准确的理解是：

- 原下跌走势类型尚未完成
- 只是其最后部分发生了更大级别的中枢扩展
- 因而当前仍处于同一个走势类型内部

后续它还可能继续演化为：

```text
a~ + A~ + b~ + B~ + c~
```

也就是在更大级别语义下，继续完成一个新的下跌走势类型。

因此，“最后一个中枢扩展”和“下跌完成后接一个更大级别盘整”在图上可能相似，但结构语义完全不同：

- 前者是未完成走势类型的延续
- 后者是两个完成走势类型的连接

#### 8.6.4 当下只能确认到哪里，就写到哪里

若当前只走到：

```text
a + A + b
```

则只能确认：

- `A` 已经是一个成立的中枢
- `b` 是离开 `A` 的后续运动
- 该运动是否最终形成与 `A` 同级别的 `B`，还不能提前认定

这时不允许因为主观预期就直接写成：

```text
a + A + b + B + c
```

因为：

- `B` 可能最终和 `A` 同级别，从而形成趋势延伸
- `B` 也可能扩展为比 `A` 更大的中枢
- 甚至 `b` 可能很快结束，使前面结构在更高层语义下被重新组合

因此项目文档、图表和解释文本都必须遵循一个原则：

- 只输出当下已经由结构确认的部分
- 对未完成部分只描述其“正在进行的语义”，不提前给完成标签

#### 8.6.5 对图表展示的直接要求

为了把上面的区别落到图表上，任何级别输出都应至少能看出：

- 最近一个已经完成的走势类型到哪里结束
- 当前未完成走势类型从哪里开始延伸
- 当前这段未完成结构是更像“同一走势类型内部扩展”，还是“前一走势完成后新类型已起步但尚未完成”

若图上只能看到局部波动，而看不出“完成边界”和“未完成边界”，则该图对缠论语义来说仍不合格。

### 8.7 趋势背驰的严格定义

本节中的“背驰”若不特别说明，默认指趋势背驰，而不是盘整背驰。

严格口径下：

- 没有趋势，没有背驰
- 趋势背驰发生在两个相邻同向趋势段的力度比较中
- 更常见也更标准的情况，是至少第二个同级别中枢之后的离开段，与前一同向离开段比较力度

用 A、B、C 三段表示时：

- A 与 C 为同向运动段
- B 为中间的连接结构，并对应一个把力度指标拉回的中间中枢或中间整理
- 若 C 段创新高或创新低，但力度弱于 A 段，则构成趋势背驰

本项目当前只先固定理论判断边界，不在本节规定唯一的程序化力度公式。可接受的辅助比较方式包括：

- 中枢语义下的结构力度比较
- 同级别 MACD 柱面积或伸长高度的辅助比较
- 均线面积等其他后续明确收敛的力度代理

但无论采用哪种代理，都必须满足：

- 比较对象是同级别、同方向的两个趋势段
- 后段在价格上创新高或创新低
- 后段力度弱于前段

### 8.8 盘整背驰的严格定义

盘整背驰不是“趋势中的标准背驰”，而是盘整语义下的一种类背驰判断。

本项目先统一以下严格口径：

- 若某级别走势尚处于第 1 个同级别中枢附近
- 某次离开中枢的运动试图脱离该中枢，但力度不足
- 随后又被拉回该中枢
- 则这种比较应归为盘整背驰，而不应归为趋势背驰

更具体地说：

- 盘整背驰常用三段结构 A、B、C 来比较第一段与第三段力度
- 这里 A 与 C 不要求都已经构成严格趋势
- 两个盘整段之间也可以比较力度
- 若 C 段突破 A 段极限位置，但力度弱于 A 段，则可认定为盘整背驰

因此：

- 第 1 个中枢之后出现的“类背驰”，默认先归入盘整背驰
- 盘整背驰的技术含义，是一次脱离中枢失败的运动
- 盘整背驰并不自动等同于大级别反转，它还可能继续演化为第三类买卖点或新的盘整连接

### 8.9 背驰段与级别收缩

若某级别某段走势构成趋势背驰或盘整背驰，则该段可称为该级别的背驰段。

后续多级别分析若要定位转折区间，应遵循：

- 先在较大级别确认背驰段
- 再到次级别中寻找该背驰段内部更精确的背驰段
- 通过级别逐步收缩，定位更小范围的转折区间

本条先作为理论与展示要求保留，当前版本不强制要求代码完成整套区间套定位。

### 8.10 图表与解释输出要求

任何级别的走势图、结构图、技术面文本或结构化输出，至少都要同时表达两部分：

- 上一个已经完成的同级别走势类型
- 当前正在进行的同级别走势类型

这样做的原因是：

- 只画当前局部，会丢掉走势类型之间的连接关系
- 只画已完成结构，会失去当下正在展开的动态语义
- 缠论分析必须同时保留“已完成的位次”和“正在进行的未完成结构”

工程上建议至少输出以下字段或等价信息：

- `last_completed_trend_type`：上一个已完成走势类型，取值为 `up`、`down`、`range` 之一
- `current_ongoing_trend_type`：当前正在进行的走势类型，若尚未确认完成则必须显式标注为 `ongoing`
- `last_completed_zs_count`：上一个已完成走势类型包含的中枢数量
- `current_structure_status`：当前结构处于延伸、盘整、离开中枢、回抽确认等哪一种状态

## 9. 买卖点口径

买卖点在第一阶段不做完整自动化定义，只做接口预留。

先统一原则：

- 买卖点必须依附于笔、线段或中枢结构
- 不能脱离结构仅根据单根 K 线给出信号
- 结果必须能回溯到触发结构

建议在第二阶段再细化：

- 一类买卖点
- 二类买卖点
- 三类买卖点
- 背驰判定
