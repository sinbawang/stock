# 基本面模块路线图 v0.1

这份文档描述基本面模块的落地顺序，避免一开始就把数据源、规则引擎、CLI、可视化、消息推送全部揉在一起。

如果阅读过程中需要回到总导航，见 [fundamental-doc-map.md](fundamental-doc-map.md)。

这份文档更适合放在“规格已经看完，准备排实现顺序”时再读。建议前置阅读：

- [fundamental-module-spec.md](fundamental-module-spec.md)
- [fundamental-v1-minimum-fields.md](fundamental-v1-minimum-fields.md)
- [fundamental-data-source.md](fundamental-data-source.md)

## 1. 总体原则

- 先文档，后实现
- 先标准模型，后具体数据源
- 先单标的分析，后批量筛选
- 先规则化评分，后复杂估值模型
- 先可解释文本输出，后图表与自动推送

## 2. 分阶段目标

### 阶段 1：规格冻结

交付：

- 基本面模块设计文档
- 行业分层规则文档
- 科技行业子模型文档
- 领域模型定义
- 打分规则
- 风险标记规则
- README 文档入口整理

完成标准：

- 对 `FundamentalSnapshot` 字段有统一定义
- 对总分、维度分、评级口径有统一定义
- 对行业分层边界有明确约定
- 对技术面与基本面的组合方式有明确边界

### 阶段 2：最小实现闭环

交付：

- 标准快照数据结构
- 评分引擎
- 文本报告输出
- 单元测试

当前仓库状态：这一阶段已经基本落地，`src/fundamental/models/`、`scoring/`、`reporting/`、`validation/`
以及对应测试已具备第一版闭环。

状态补充：

- 文本报告当前已不只输出“总分 + 评级 + 优势风险”
- 还会输出维度得分的简版计算依据；按当前第一版实现，维度内仍是“已计分子项做平均，再乘维度权重”
- 对港股金融 live 字段，还会输出字段来源相关的口径警告

评分结构补充：

- 当前第一版评分引擎只有“维度权重”，还没有“维度内子项配额/子项权重”这一层配置
- 这使得当前实现更容易扩模型、统一解释，但维度内部的重要性层次表达仍然偏弱
- 后续建议把核心维度逐步从“维度内平均制”演进到“分项配额制”
- 优先改造的维度应包括：`profit_quality`、`capital_safety_and_asset_quality`、`profitability_and_stability`、`valuation_fit`
- 改造目标不是推翻维度权重，而是在保留维度权重的前提下，为维度内关键指标再增加一层显式配额

完成标准：

- 输入一份标准 JSON 快照
- 输出总分、评级、优势、风险、缺失项
- 对异常字段和缺失字段有稳定处理

### 阶段 3：数据源接入

优先级建议：

1. 手工 JSON / CSV 导入
2. `akshare`
3. `tushare`
4. 其他外部源

完成标准：

- 至少一个数据源能稳定产出标准快照
- 数据源层和评分层解耦

当前仓库状态：这一阶段也已进入已落地状态，港股与 A 股公共快照入口已经分别沉到
`src/fundamental/data/hk_snapshot_fetcher.py` 与 `src/fundamental/data/cn_snapshot_fetcher.py`，并通过
`src/fundamental/services/` 提供抓取并分析入口。

补充说明：

- A 股银行 live 链路已经可跑通，`601328` 当前可通过公共源抓取并进入 `bank_v1` 分析
- 银行专属字段当前主要来自 Eastmoney A 股主指标表，例如 `NEWCAPITALADER`、`HXYJBCZL`、`NONPERLOAN`、`BLDKBBL`、`NET_INTEREST_MARGIN`
- 当前 THS A 股摘要表 schema 已经变化，代码侧已改为“列名优先，列位兜底”的解析方式
- A 股银行 `dividend_yield` 暂无稳定点位公共源，因此当前在 live 分析链路里按运行时放宽处理
- 港股保险 / 券商的公共源仍只稳定提供通用盈利、估值、股息率等字段，尚未稳定提供 `solvency_adequacy_ratio`、`embedded_value_growth`、`new_business_value_growth`、`net_capital_ratio` 等监管型字段
- 因此 `insurance_v1` 与 `broker_v1` 的第一版现实路径已经变成：
	- 主源继续提供通用财务字段
	- 关键监管字段优先走公司官方披露 fallback
	- 剩余无法自动化的字段保留 `manual supplement`

阶段 3 的最新状态可以更具体地写成：

- `06886` 已能通过华泰官方年报 fallback 补齐 `net_capital_ratio` 代理值，并进入 `broker_v1` live 分析
- `01339` 已能通过人保官网偿付能力披露 fallback 补齐 `solvency_adequacy_ratio`
- 但 `insurance_v1` 其余字段如 `combined_ratio`、`investment_return`、`embedded_value_growth`、`new_business_value_growth` 仍未完全自动化，因此人保 live 完整报告仍主要依赖 `manual supplement`

### 阶段 4：与技术面联动

交付：

- 联合分析结果模型
- 技术面结果与基本面评分的整合逻辑

完成标准：

当前仓库状态：这一阶段也已进入已落地状态，联合文本报告和微信群发送链路已经可用，并且：

- 联合报告会保留基本面 `警告` 段，提示字段来源口径
- 联合报告里的维度得分会附简版计算说明，减少“这个分是怎么算出来的”这类解释成本

### 阶段 5：批量分析与输出

交付：

- 汇总表输出

## 3. 当前模块分层

按当前仓库实现，基本面代码已经大体分为七层：

- `data`: 财报或指标数据获取，以及不同来源字段到标准快照的映射
- `models`: 标准快照与评分输出模型
- `config`: 子模型配置对象与注册表
- `validation`: 字段完整性与准入校验
- `scoring`: 基本面评分与风险标记
- `reporting`: 文本报告输出
- `services`: “抓取 + 分析”总入口

这比早期草案里的 `providers / normalizers / scoring / reporting` 更贴近当前实现，也更符合现在的导出接口。

## 4. 第一阶段必须避免的事情

- 把抓数逻辑和评分逻辑写在同一个脚本里
- 直接把某个网站的字段命名当成领域模型
- 没有定义缺失值规则就开始写 if/else
- 总分规则没冻结就开始做联合筛选
- 没有测试样例就引入大量外部依赖

## 5. 待确认问题

这些问题在继续扩模型和联动前仍然值得明确：

- 港股和 A 股是否使用完全相同的评分阈值
- 科技行业是否先只覆盖平台互联网和半导体
- 工业自动化与游戏内容是否在第一阶段一并纳入科技扩展模型
- 能源资源行业是否先只覆盖煤炭、石油石化、有色
- `pb`、`ps_ttm` 是否作为通用核心指标，还是只保留在估值扩展层
- 报告期冲突时，是否允许年报和 TTM 混合
- 多期趋势评分是否放到第二阶段还是第三阶段
- 当前评分结构是否继续维持“维度内平均制”，还是逐步升级到“分项配额制”

已明确事项：

- 金融股已经单独拆出子模型，而不是继续套用通用科技/制造口径
- 第一版金融子模型已经包括：`bank_v1`、`insurance_v1`、`broker_v1`
- 金融行业第一阶段的主要剩余问题不再是“要不要单独建模”，而是“如何补齐 live 数据源字段”
- 当前仓库中的真实实现仍是“维度内平均制”；`分项配额制` 被视为下一阶段评分引擎升级方向，而不是已落地能力

## 6. 建议的实现顺序

建议严格按这个顺序推进：

1. 冻结文档
2. 冻结行业分层边界
3. 确认示例输入 JSON
4. 实现 `FundamentalSnapshot`
5. 先实现科技行业主模型：平台互联网 + 半导体硬科技
6. 再实现科技扩展模型：工业自动化 + 游戏内容
7. 补测试
8. 再接数据源
9. 实现金融行业子模型：银行 + 保险 + 券商
10. 修通 live 数据源字段映射，优先 A 股银行，再补港股保险 / 券商第二数据源
11. 扩展能源资源
12. 最后再做与缠论主流程联动

按当前仓库状态，这个顺序已经推进到第 10 步的中段：

- 科技四个子模型已落地
- 金融三类子模型已落地
- A 股银行 live 数据源已打通第一版
- 港股保险 / 券商仍需要第二数据源或更细粒度公开指标表支持

截至 2026-05-11，当前更准确的描述是：

- 港股保险 / 券商第二数据源已经不再是纯规划状态
- 华泰券商 live 已实现“官方年报 fallback + 报告口径警告”的第一版闭环
- 人保保险 live 已实现“偿付能力官方披露 fallback + 其余字段手工补充 + 报告口径警告”的半自动闭环
- 当前主要收尾方向不再是“能不能先跑起来”，而是两条并行线：
	- 还能自动化掉哪些剩余手工字段，以及如何在报告里继续降低口径歧义
	- 如何把当前评分引擎从“维度内平均制”升级到更符合业务主次的“分项配额制”
