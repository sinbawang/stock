# 基本面模块路线图 v0.1

这份文档描述基本面模块的落地顺序，避免一开始就把数据源、规则引擎、CLI、可视化、消息推送全部揉在一起。

## 1. 总体原则

- 先文档，后实现
- 先标准模型，后具体数据源
- 先单标的分析，后批量筛选
- 先规则化评分，后复杂估值模型
- 先可解释文本输出，后图表与自动推送

## 2. 分阶段目标

### 阶段 1：规格冻结

交付：

- 基本面模块设计文档
- 行业分层规则文档
- 科技行业子模型文档
- 领域模型定义
- 打分规则
- 风险标记规则
- README 文档入口整理

完成标准：

- 对 `FundamentalSnapshot` 字段有统一定义
- 对总分、维度分、评级口径有统一定义
- 对行业分层边界有明确约定
- 对技术面与基本面的组合方式有明确边界

### 阶段 2：最小实现闭环

交付：

- 标准快照数据结构
- 评分引擎
- 文本报告输出
- 单元测试

完成标准：

- 输入一份标准 JSON 快照
- 输出总分、评级、优势、风险、缺失项
- 对异常字段和缺失字段有稳定处理

### 阶段 3：数据源接入

优先级建议：

1. 手工 JSON / CSV 导入
2. `akshare`
3. `tushare`
4. 其他外部源

完成标准：

- 至少一个数据源能稳定产出标准快照
- 数据源层和评分层解耦

### 阶段 4：与技术面联动

交付：

- 联合分析结果模型
- 技术面结果与基本面评分的整合逻辑
- 候选池过滤规则

完成标准：

- 可以对已有缠论分析结果追加基本面评级
- 可以标记“技术面成立但基本面不支持”的情况

### 阶段 5：批量分析与输出

交付：

- 批量输入
- 汇总表输出
- 与现有报告导出链路结合

## 3. 模块分层建议

建议未来代码分为四层：

- `providers`: 财报或指标数据获取
- `normalizers`: 不同来源字段映射到标准快照
- `scoring`: 基本面评分与风险标记
- `reporting`: 文本报告、CSV 输出、后续可视化

这样可以避免后续改一个数据源时，评分逻辑也跟着震荡。

## 4. 第一阶段必须避免的事情

- 把抓数逻辑和评分逻辑写在同一个脚本里
- 直接把某个网站的字段命名当成领域模型
- 没有定义缺失值规则就开始写 if/else
- 总分规则没冻结就开始做联合筛选
- 没有测试样例就引入大量外部依赖

## 5. 待确认问题

这些问题在开始实现前应明确：

- 港股和 A 股是否使用完全相同的评分阈值
- 金融股是否需要单独评分模型
- 科技行业是否先只覆盖平台互联网和半导体
- 工业自动化与游戏内容是否在第一阶段一并纳入科技扩展模型
- 能源资源行业是否先只覆盖煤炭、石油石化、有色
- `pb`、`ps_ttm` 是否作为通用核心指标，还是只保留在估值扩展层
- 报告期冲突时，是否允许年报和 TTM 混合
- 多期趋势评分是否放到第二阶段还是第三阶段

## 6. 建议的实现顺序

建议严格按这个顺序推进：

1. 冻结文档
2. 冻结行业分层边界
3. 确认示例输入 JSON
4. 实现 `FundamentalSnapshot`
5. 先实现科技行业主模型：平台互联网 + 半导体硬科技
6. 再实现科技扩展模型：工业自动化 + 游戏内容
7. 补测试
8. 再接数据源
9. 扩展能源资源
10. 最后再做金融行业与缠论主流程联动
